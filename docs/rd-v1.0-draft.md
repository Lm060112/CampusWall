# 校园生活小程序 V1.0 上线版 RD 草案

> 文档状态：草案  
> 适用阶段：从当前前端模拟版推进到校内小范围真实试运行版  
> 当前版本：V1.0 Draft  
> 说明：本文档不是最终定稿，其中标注“待确认”的内容需要和项目负责人、协作者、学校/商家/运营人员进一步确认。

## 1. 项目背景

校园生活小程序希望为学生提供一站式校园生活服务，覆盖校内点单、校外外卖、跑腿代取、周边玩乐、校园广场、消息通知和个人中心等场景。

当前项目已经完成前端主要页面和本地模拟闭环，但还没有正式接入云数据库、云函数、支付、真实商家和真实运营后台。

V1.0 的目标不是一次性做成完整商业平台，而是做出一个可以在校园内小范围试运行的版本。

## 2. V1.0 产品目标

### 2.1 核心目标

V1.0 需要实现：

- 学生可以浏览校园生活服务。
- 学生可以完成校内点单、外卖下单、跑腿需求、活动预约等基本流程。
- 用户数据、订单数据、地址数据、消息数据从本地缓存迁移到云数据库。
- 订单状态可以被可信地创建、更新、查询。
- 管理员或服务方可以处理基础订单。
- 用户可以在“我的订单”和“消息”里跟踪服务状态。
- 小程序具备小范围真实试运行能力。

### 2.2 非目标

V1.0 暂不追求：

- 大规模商家入驻。
- 完整配送调度系统。
- 完整财务结算系统。
- 复杂优惠券和营销系统。
- 完整商家 SaaS 后台。
- 全国/全校大范围上线。

## 3. 用户角色

### 3.1 学生用户

主要使用者。

能力：

- 浏览首页和服务模块。
- 下单/预约/发布跑腿需求。
- 查看订单。
- 管理地址。
- 浏览和发布校园广场内容。
- 接收消息通知。
- 收藏、评论、点赞。

### 3.2 校内服务方

例如校内咖啡厅、食堂窗口、便利服务点。

能力：

- 查看本服务点订单。
- 修改订单状态。
- 维护基础商品信息。

待确认：

- V1.0 是否需要单独的服务方登录入口。
- 服务方是否直接使用小程序端管理订单，还是由管理员代操作。

### 3.3 外卖商家

校外商家。

能力：

- 展示商家信息。
- 展示商品菜单。
- 接收订单。

待确认：

- V1.0 是否真实接入校外商家。
- 外卖是否由商家配送、学生自取，还是只做信息展示和预约。

### 3.4 跑腿员

可以接跑腿订单的人。

能力：

- 查看可接跑腿订单。
- 接单。
- 更新订单状态。

待确认：

- 是否允许普通学生成为跑腿员。
- 是否需要身份认证。
- 是否需要押金/信用机制。
- 是否需要学校审核。

### 3.5 管理员

项目运营和审核人员。

能力：

- 管理用户。
- 管理商家/服务点。
- 查看和处理订单。
- 审核校园广场内容。
- 管理举报。
- 处理异常订单。

## 4. 核心模块

## 4.1 首页

### 功能说明

首页是校园生活服务的主要入口。

包含：

- 校区展示。
- 搜索入口。
- 四大服务入口：
  - 校内点单
  - 校外外卖
  - 跑腿代取
  - 周边玩乐
- 热门推荐。
- 附近商家/服务点。
- 校园公告。

### V1.0 要求

- 首页服务入口必须能进入对应模块。
- 推荐内容应从云数据库读取。
- 无数据时显示空状态或默认推荐。
- 搜索入口可以先支持基础搜索。

待确认：

- 首页推荐规则是手动配置还是按销量/热度排序。
- 是否按校区展示不同内容。

## 4.2 校内点单

### 场景说明

面向校内咖啡厅、食堂窗口、轻食窗口、便利点等校园服务点。

目标流程：

```text
首页 → 校内点单 → 服务点列表 → 服务点菜单 → 加购 → 确认点单 → 提交订单 → 服务点处理 → 用户取餐 → 订单完成
```

### V1.0 功能

- 服务点列表。
- 服务点分类。
- 服务点营业状态。
- 菜单展示。
- 商品加购。
- 商品规格选择，至少支持备注。
- 购物车。
- 确认订单。
- 取餐号。
- 订单状态：
  - 制作中
  - 待取餐
  - 已完成
  - 已取消

### 后端要求

需要集合：

- `merchants`
- `products`
- `orders`
- `messages`

需要云函数：

- `createOrder`
- `updateOrderStatus`
- `createMessage`

待确认：

- 校内点单是否需要在线支付。
- 如果不支付，是否到店付款。
- 服务点是否自己更新订单状态。

  - 商家备餐

### 后端要求

## 6. 云函数设计

V1.0 至少需要：

| 云函数 | 说明 |
| --- | --- |
| `login` | 获取 openid，初始化用户 |
| `createOrder` | 创建订单 |
| `updateOrderStatus` | 更新订单状态 |
| `createMessage` | 写入消息 |
| `saveAddress` | 保存地址 |
| `createPost` | 发布帖子 |

| `likePost` | 点赞/取消点赞 |
| `addComment` | 添加评论 |

- 订单金额必须由后端根据商品价格计算，不能完全相信前端传入金额。

- 用户只能读取自己的订单、地址、消息。
- 管理员和商家权限需要单独校验。

## 8. 支付方案

V1.0 有两种方案。

### 方案 A：不接支付，线下结算

优点：

- 上线快。
- 技术复杂度低。
- 审核和资质压力小。
- 适合校内小范围试运行。

缺点：

- 无法线上闭环付款。
- 订单爽约风险高。
- 需要人工确认。

### 方案 B：接微信支付

优点：

- 流程完整。
- 可做退款、优惠券、财务统计。

缺点：

- 需要商户号。
- 需要后端签名。
- 需要支付安全校验。
- 涉及退款、对账、资金风险。

V1.0 建议：

```text
优先采用方案 A，先做校内小范围试运行。
支付能力放到 V1.1 或 V2.0。
```

待确认：

- 是否必须接微信支付。
- 学校/商家是否接受线下付款。

## 15. 风险

| 风险 | 说明 | 建议 |
| --- | --- | --- |
| 数据风险 | 当前本地模拟数据不能真实运营 | 尽快迁移云数据库 |
| 支付风险 | 支付、退款、对账复杂 | V1.0 暂缓支付 |
| 履约风险 | 外卖/跑腿真实服务涉及纠纷 | 先小范围试运行 |
| 内容风险 | 发现页可能出现违规内容 | 接内容审核和举报 |
| 权限风险 | 用户可能伪造前端请求 | 关键逻辑放云函数 |
| 运营风险 | 商家和跑腿员需要管理 | 先做简版管理端 |

## 16. 推荐 V1.0 路线

推荐路线：

```text
当前前端模拟版
→ 补齐外卖规格和购物车
→ 云数据库 orders / addresses / messages
→ 云函数 createOrder / updateOrderStatus
→ 发现页详情和内容审核
→ 简版管理员订单处理
→ 小范围试运行
```

不建议一开始就做：

- 微信支付。
- 大量真实商家入驻。
- 完整跑腿员体系。
- 复杂优惠券系统。

## 17. 结论

当前项目已经具备产品原型和主要前端流程。

V1.0 的重点不是继续堆页面，而是：

- 把本地模拟数据迁移到云数据库。
- 把关键业务逻辑迁移到云函数。
- 建立基础权限和审核机制。
- 做小范围真实试运行。

只有完成这些，小程序才算从“演示版”进入“真实可上线版本”。
