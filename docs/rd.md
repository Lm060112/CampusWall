# 校园生活小程序 RD

> 面向协作者的需求与研发说明。本文档用于快速了解项目定位、当前进度、运行方式、模块边界和后续开发计划。

## 1. 项目定位

本项目是一个面向校园场景的微信小程序，目标是整合校园生活服务与校园社区内容。

当前产品方向不是单纯“校园墙”，而是“校园生活服务平台”：

- 首页：校园生活服务入口与推荐信息
- 校内点单：校内咖啡厅、食堂窗口等服务点在线点单，到店自取
- 校外外卖：校外商家列表、进店点单、配送到校门/宿舍区
- 跑腿代取：取快递、代买、代送、打印取件等校园跑腿需求
- 周边玩乐：活动、运动、桌游、探店等预约
- 发现：校园广场/校园墙，支持发布、点赞、评论等社区互动
- 消息：订单消息、互动消息、系统通知
- 我的：订单、地址、收藏、发布、设置等个人中心

## 2. 当前技术栈

当前是原生微信小程序项目。

主要技术：

- 微信小程序原生 `WXML / WXSS / JS / JSON`
- 微信云开发能力，当前已有云函数目录
- Vant Weapp 组件库
- 本地缓存模拟数据：`wx.getStorageSync` / `wx.setStorageSync`
- GitHub 仓库：`https://github.com/Lm060112/CampusWall.git`

当前核心流程大多仍是“前端页面 + 本地模拟数据”，还没有全面接入正式云数据库。

## 6. 当前模拟数据说明

当前本地模拟数据主要保存在微信小程序本地缓存：

| Key | 说明 |
| --- | --- |
| `mockOrders` | 本地模拟订单列表 |
| `mockMessages` | 本地模拟消息 |
| `mockDefaultAddress` | 默认收货地址 |
| `userInfo` | 模拟用户信息 |

注意：

- 本地缓存不是正式数据库
- 清除缓存后数据会丢失
- 不同用户/不同设备之间数据不同步
- 后续上线前必须迁移到云数据库和云函数

## 7. 当前运行方式

1. 使用微信开发者工具打开项目根目录 `D:\CampusWall`
2. 确认小程序项目配置读取 `project.config.json`
3. 如果 Vant 组件异常，先执行小程序 npm 构建

## 8. 协作开发约定

### 8.1 分支建议

不要多人直接改 `main`。

建议：

```text

feature/discover-detail      发现详情
```

### 8.2 提交建议

提交信息尽量写清楚模块：

```text
feat(order): 接入云数据库订单创建
feat(discover): 新增帖子详情页
fix(message): 修复订单消息未读数量
docs: 更新协作 RD
```

### 8.3 开发前需要确认

每个协作者开工前需要确认：

- 改哪个模块
- 是否涉及公共订单结构
- 是否涉及 `app.json` 路由
- 是否涉及云数据库集合
- 是否会影响现有本地 mock 数据

## 9. 下一阶段优先级

### P0：先完成，适合当前阶段

这些是前端和模拟流程层面的增强，现阶段可以继续做。

1. 商品规格选择
   - 奶茶冰量、糖度、小料
   - 餐品口味、辣度、打包选项

2. 购物车弹层
   - 展开已选商品
   - 修改数量
   - 清空购物车
   - 展示配送费/包装费/优惠

3. 外卖筛选排序
   - 综合排序
   - 销量
   - 配送最快
   - 起送价
   - 优惠

4. 多地址管理
   - 地址列表
   - 新增/编辑/删除
   - 默认地址

5. 订单取消和评价
   - 未完成订单可取消
   - 已完成订单可评价
   - 本地模拟评价数据

### P1：需要开始接云开发

这些是从“前端模拟”进入“真实后端”的关键。

1. 云数据库集合
   - `users`
   - `addresses`
   - `merchants`
   - `products`
   - `orders`
   - `messages`
   - `posts`
   - `comments`
   - `favorites`
   - `coupons`

2. 云函数
   - `createOrder`
   - `updateOrderStatus`
   - `createMessage`
   - `saveAddress`
   - `createPost`
   - `addComment`
   - `likePost`
   - `addFavorite`

3. 权限校验
   - 用户只能看自己的订单
   - 用户只能改自己的地址
   - 用户不能伪造订单金额
   - 管理员/商家权限单独区分

### P2：上线前必须补齐

1. 登录与用户身份
2. 隐私协议/用户协议
3. 内容安全审核
4. 真机测试
5. 空状态、加载态、失败态
6. 云环境配置
7. 数据备份和基础风控

### P3：暂不建议马上做

1. 微信支付
2. 正式退款
3. 真实商家入驻
4. 跑腿员实时抢单
5. 多角色后台完整权限系统

这些需要更完整的后端、运营规则和资质准备。

## 12. 当前核心风险

1. 数据仍以本地缓存为主，不能支持真实多用户。
2. 订单金额、状态变化都在前端，正式上线前必须移到云函数校验。
3. 发现页涉及用户内容，正式上线前必须接内容安全审核。
4. 校外外卖、跑腿、活动都涉及真实履约，正式运营需要规则和人员处理。
5. 当前 UI 已有较多页面，后续要注意统一组件和样式，避免重复代码越来越多。
